# 多人并行修改与合并说明

## 1. 这份文档解决什么问题

你们现在是多人并行给同一个前端项目加功能，最后还要把功能合到一起。  
本说明的目标是两件事：

1. 降低多人同时改同一文件时的冲突概率。
2. 让按钮的说明反馈与模型能力走统一入口，而不是每个人各写一套。

## 2. 先确认仓库状态

当前这个目录里没有 `.git`，说明你现在看到的工作目录本身还不是一个 Git 仓库。

如果你们后面要多人协作，建议二选一：

1. 在真正的仓库根目录里继续开发。
2. 如果这就是最终项目目录，就先初始化 Git：

```bash
git init
git branch -M main
git remote add origin <你们的远端仓库地址>
```

然后再开始下面的分支协作流程。

## 3. 当前项目里和说明反馈/模型调用相关的核心结构

这次已经把“按钮说明接到统一能力层”收敛好了，核心文件如下：

- `src/lib/aiButtonAssistant.ts`
  统一负责按钮点击后的说明生成、JSON 解析和本地兜底文案。
- `src/components/AiButtonAssistantProvider.tsx`
  全局监听按钮点击，自动把当前页面、按钮语义和用户状态送进统一说明层。
- `src/components/AiAssistantDock.tsx`
  页面右下角的提示面板，展示返回的解释。
- `src/App.tsx`
  已接入全局 `AiButtonAssistantProvider`。

已经存在的模型能力主链路仍然保留：

- `src/lib/scheduleImport.ts`：课表 OCR
- `src/lib/sleepScheduler.ts`：时间轴补觉规划

## 4. 以后新增按钮，正确接统一说明层的方式

以后任何人新增按钮，优先遵循这个规则：

1. 用真正的 `<button>`，不要继续用 `div onClick` 模拟按钮。
2. 给按钮补语义字段，至少补一个：
   - `data-ai-label`
   - `data-ai-context`
3. 原有业务逻辑照常写在 `onClick` 里，不要自己再单独写一套说明请求。

推荐写法：

```tsx
<button
  type="button"
  onClick={handleSomething}
  data-ai-label="生成今晚补觉建议"
  data-ai-context="这个按钮会结合今日欠眠、课表和路线偏好给出解释"
>
  查看建议
</button>
```

说明：

- `data-ai-label`：告诉全局说明层“这个按钮叫什么”。
- `data-ai-context`：告诉全局说明层“这个按钮为什么会被点、会影响什么”。
- 如果不写，系统也会尽量从按钮文本、`title`、`aria-label` 里猜，但效果会差一些。

## 5. 推荐分工边界

为了少冲突，建议按“页面/能力边界”分工，不要多人同时大改一个文件。

建议边界：

- A 同学：`src/pages/Onboarding.tsx`、OCR 校准相关
- B 同学：`src/pages/Home.tsx`、`src/components/SleepMode.tsx`
- C 同学：`src/pages/Timeline.tsx`、`src/lib/sleepScheduler.ts`
- D 同学：`src/pages/Map.tsx`、地图与路线能力
- E 同学：`src/pages/Profile.tsx`、偏好设置与用户信息
- 一个人统一维护：`src/lib/aiButtonAssistant.ts` 与 `src/components/AiButtonAssistantProvider.tsx`

不要多人同时重写这些高冲突文件：

- `src/App.tsx`
- `src/store.tsx`
- `src/index.css`
- `src/components/AiButtonAssistantProvider.tsx`
- `src/lib/aiButtonAssistant.ts`

如果必须一起改，先约定负责人，其他人只提局部补丁。

## 6. 每个人日常开发流程

### 开新分支

```bash
git checkout main
git pull origin main
git checkout -b feat/<你的功能名>
```

例子：

```bash
git checkout -b feat/profile-ai-settings
git checkout -b feat/map-ai-recommendation
```

### 开发时的最小原则

1. 优先新增组件/工具文件，少在一个超大页面里堆逻辑。
2. 如果只是想让新按钮带出更完整的说明，不要自己写新 fetch，直接补 `data-ai-*`。
3. 页面里如果出现重复提示块，优先抽到组件。
4. 如果要改样式，尽量只动自己负责页面附近的结构。

### 提交前自查

```bash
npm run lint
npm run build
```

通过后再提交：

```bash
git add .
git commit -m "feat: add ai enhancements for profile settings"
```

## 7. 合并前的同步流程

每天至少做一次：

```bash
git checkout main
git pull origin main
git checkout feat/<你的功能名>
git merge main
```

如果你们团队更喜欢线性历史，也可以改成：

```bash
git checkout feat/<你的功能名>
git rebase main
```

但如果团队里有人不熟悉 rebase，统一用 `merge main` 更稳。

## 8. 冲突时怎么处理

优先按这个顺序看：

1. 看是不是大家都在改同一个页面文件。
2. 看是不是有人直接在页面里各自写了独立说明请求。
3. 看是不是有人把 `div onClick` 和 `button` 改法混在一起。
4. 看是不是 `data-ai-label`、`data-ai-context` 被覆盖掉了。

解决原则：

1. 保留原来的业务点击逻辑。
2. 保留这次统一的全局按钮说明入口，不要回退成每页单独请求。
3. 如果两个分支都给同一个按钮补了说明语义，优先保留更具体的 `data-ai-label` / `data-ai-context`。
4. 冲突解决后，必须重新跑：

```bash
npm run lint
npm run build
```

## 9. 合并这套“统一按钮说明层”时要特别注意什么

### 不要重复造轮子

已经有统一入口：

- `AiButtonAssistantProvider`
- `aiButtonAssistant.ts`

以后新增按钮说明，只做语义补充，不要再新增：

- 第二个全局监听器
- 第二套说明弹窗
- 第二套 OpenAI-compatible fetch 封装

### 新功能如果需要单独的模型能力

如果不是“按钮解释”，而是实打实的新能力，例如：

- 智能推荐休息地点
- 分析一周睡眠趋势
- 生成个性化睡前计划

可以单独建新的 `lib` 文件，但要和按钮说明层分开。

建议命名：

- `src/lib/aiRestPointRecommendation.ts`
- `src/lib/aiSleepTrendSummary.ts`

按钮只负责“触发”，能力本身放在单独文件里。

## 10. 合并完成后的验收清单

主流程至少点一遍：

1. 引导页上传课表按钮是否还能触发 OCR
2. OCR 校准页的新增/删除/保存按钮是否都还能用
3. 首页的记录睡眠、开始小睡、目标时间按钮是否正常
4. 时间轴里的说明按钮和开启睡眠模式是否正常
5. 地图页路线说明/休息建议按钮是否能弹出提示面板
6. 个人页的设置项、编辑资料、退出登录是否还能点击
7. 右下角提示面板是否会随着按钮点击更新内容

## 11. 现在最适合你们团队的协作约定

建议直接统一成一句话：

“业务逻辑各做各的，按钮说明统一走全局层，新按钮必须写 `data-ai-label` 和 `data-ai-context`。”  

这样后面谁加功能，谁都知道该往哪里接，不会越做越乱。
